# 📐 Northstar Style Guide — Client-Facing Document Standards

> **Authority level:** MANDATORY. Every agent (Antigravity, Codex, or any other) MUST read this file and follow it exactly before generating any client-facing document.
> **Last updated:** 2026-02-18

---

## 1 · Why This File Exists

The client pays for clarity, not raw data. Every document that leaves this agency must feel like it was written by a senior recruitment consultant — not pasted from a terminal. If a document looks like a debug log, it has **failed** and must be rewritten.

## 2 · Voice & Tone

| Do | Don't |
|:---|:---|
| Sound like a committed sourcing agent talking to a real person | Sound like a generic AI assistant |
| Be direct, calm, practical, and supportive | Be robotic, repetitive, or overly formal |
| Use short declarative sentences | Use filler phrases ("As an AI…", "Here is the data…") |
| Explain what data *means* for the client | Dump raw numbers without interpretation |
| Acknowledge urgency without panic | Use alarming language or false hype |

**Golden rule:** If you remove all the tables from the document, the remaining prose should still make sense and still sound like a human recruiter wrote it.

## 3 · Document Anatomy (Required Structure)

Every client-facing report MUST follow this skeleton. Sections may be skipped only if explicitly marked optional.

```
# 🏆 [Document Title] — [Subtitle]

> **Prepared by** Northstar Job Sourcing Desk · **Date** [DD Month YYYY]
> **Market** [Market] · **Focus** [Focus description]

---

## 1 · [Executive Snapshot Section]
  - KPI table or metric block
  - 1-3 line "What this means →" interpretation

---

## 2 · [Primary Data Section]
  - Main table (winners, priorities, etc.)
  - Optional: "Why these stand out" narrative block

---

## 3 · [Secondary Data Section]
  - Backup/supporting table
  - Optional: Quick tip callout

---

## 4 · [Delta / Change Section] (optional)

  - Source file reference table

---

<sub>Generated by Northstar Job Sourcing Desk · [metadata]</sub>
```

## 4 · Formatting Rules (Non-Negotiable)

### 4.1 · Headings

- **H1** (`#`): Document title only. ONE per document. Include a relevant emoji.
- **H2** (`##`): Major sections. Use numbered format: `## 1 · Section Name`
- **H3** (`###`): Sub-sections within a major section. Use for narrative blocks like "Why these stand out".
- Never skip heading levels (no H1 → H3 without H2).

### 4.2 · Header Block

Every document starts with a blockquote metadata bar:

```markdown
> **Prepared by** Northstar Job Sourcing Desk · **Date** 16 February 2026
> **Market** Portugal · **Focus** L1 IT / Helpdesk / Service Desk
```

### 4.3 · Section Separators

Use `---` (horizontal rule) between every major H2 section. This creates visual breathing room.

### 4.4 · Tables

| Rule | Example |
|:---|:---|
| Always include a sequential `#` column for numbered rows | `| 1 |`, `| 2 |` |
| Center-align `#`, Tier, and icon columns | `:---:` |
| Right-align numeric Score columns | `---:` |
| Left-align Company, Role, and text columns | `:---` |
| **Links must be short markdown hyperlinks** | `[Apply →](url)` or `[View →](url)` |
| **NEVER paste raw URLs into table cells** | ❌ `https://www.linkedin.com/jobs/view/...` |
| Bold the top-priority rows | `| **CodeWin** | **Service Desk** |` |
| Use ✅ / ❌ icons for boolean columns (Easy Apply) | Not "Yes" / "No" text |

**Example of a correct table:**

```markdown
| # | Company | Role | Tier | Score | Easy Apply | Action |
|:---:|:---|:---|:---:|---:|:---:|:---|
| 1 | **CodeWin** | Service Desk Specialist | **A** | **18** | ✅ Yes | [**Apply →**](https://linkedin.com/jobs/view/123/) |
| 2 | **Real Hotels Group** | Service Desk | **A** | **18** | ❌ No | [**Apply →**](https://linkedin.com/jobs/view/456/) |
```

**Example of an INCORRECT table (DO NOT produce this):**

```markdown
| Priority | Company | Job Title | Tier | Score | Easy Apply | Link |
| :--- | :--- | :--- | :--- | ---: | :--- | :--- |
| 1 | CodeWin | Service Desk Specialist | A | 18 | Yes | https://www.linkedin.com/jobs/view/4372570740/?alternateChannel=search&eBP=... |
```

### 4.5 · Interpretation Blocks

After every data table or metric block, include a short **"What this means →"** paragraph. This is what separates a professional brief from a raw dump.

```markdown
**What this means →** Today's sweep returned two high-confidence Service Desk roles.
Both scored 18 points — the highest possible this cycle. The pipeline is healthy.
*This is a "go apply" day.*
```

### 4.6 · Callout Blocks

Use blockquotes (`>`) for executive summaries and key takeaways:

```markdown
> **Good news:** Your case has moved from exploration to execution mode.
> You have 2 immediate A-tier targets and 11 backup roles ready to apply.
```

Use the `> **Quick tip:**` pattern for inline tips:

```markdown
> **Quick tip:** Rows marked ✅ (Easy Apply) can be submitted in under a minute each.
```

### 4.7 · Risk / Note Items

Use emoji-prefixed bullet points:

```markdown
- ⚠️ **Language mismatch** — Several B-tier listings are in Portuguese only.
- ⚠️ **Role ambiguity** — Linde title suggests physical maintenance, not IT.
- ✅ **No expired links detected** — All 13 listings were live at scraping time.
```

### 4.8 · Execution Plan Format

Use a numbered table for action plans, not a plain list:

```markdown
| Step | Action | Target |
|:---:|:---|:---|
| **1** | **Apply to both A-tier winners** | CodeWin + Real Hotels Group |
| **2** | **Batch-apply Easy Apply B-tier** | Up to 7 quick applications |
| **3** | **Log results** | Update CASE_LOG.md |
```

End with a realistic estimate:

```markdown
**Realistic output:** If you follow this plan, you can submit **10+ applications within 60 minutes**.
```

### 4.9 · Footer

Every document ends with a subtle metadata line:

```markdown
<sub>Generated by **Northstar Job Sourcing Desk** · Run ID: [IDs] · [date]</sub>
```

### 4.10 · Links

- **In tables:** Always use `[Apply →](url)` or `[View →](url)`.
- **In prose:** Use `[descriptive text](url)`.
- **Strip tracking parameters** from LinkedIn URLs when possible. Use the short form: `https://www.linkedin.com/jobs/view/[JOB_ID]/`
- **NEVER** paste a raw URL longer than ~60 characters into any visible position.

## 5 · Document Templates

### 5.1 · WINNER_REPORT_TODAY.md

**Purpose:** Daily winners brief. The client opens this to decide what to apply to.

**Required sections:**
1. Daily Outcome Snapshot (metric table + interpretation)
2. Top Priority Jobs — Apply First (numbered table with hyperlinks)
3. "Why these stand out" narrative (short per-job commentary)
4. Backup Pipeline — Apply Next (numbered table continuing sequence)
5. What Changed Since Last Run (or "No prior delta available")
6. Execution Plan — Next 60 Minutes (action table)
7. Risks & Notes (emoji-prefixed bullets)
8. Data Sources (reference table)

### 5.2 · CLIENT_STATUS_APPOINTMENT_BRIEF

**Purpose:** Full case status update. Like a recruiter calling the client with a progress report.

**Required sections:**
1. Client info header block
2. Executive status summary (blockquote)
3. What the Agency has done (numbered narrative blocks with context)
4. Current pipeline snapshot (KPI table)
5. Priority Board (A-tier table with hyperlinks)
6. Backup Pipeline (B-tier table)
7. Client-friendly interpretation ("What This Means For You")
8. 60-Minute Execution Plan
9. Case outlook / next milestone
10. Source integrity / data provenance

### 5.3 · Market Density Report

**Purpose:** Market analysis for a geographic/keyword segment.

**Required sections:**
1. Market overview (density classification)
2. Keyword performance matrix
3. Signal rate analysis
4. Recommendation (keep/retire/expand)
5. Data sources

### 5.4 · A-Tier Fit Validation Dossier

**Purpose:** Prove to the client why a top-tier role is genuinely tailored before recommending application.

**Required sections:**
1. Role snapshot (company, role, tier, score, apply link)
2. Why ranked A-tier (scoring logic translated to plain language)
3. Requirement-to-client match matrix (listing evidence vs client evidence)
4. Risk and confidence audit
5. Application readiness gate (`GO` / `CONDITIONAL GO` / `HOLD`)
6. Execution support package (CV bullets, message template, follow-up plan)
7. Data provenance table

### 5.5 · Application Packet

**Purpose:** Convert GO/CONDITIONAL GO dossiers into execution-ready assets.

**Required sections:**
1. Role summary (company, role, link, tier)
2. Top 3 extracted requirements
3. Requirement-to-evidence mapping table
4. 3 tailored CV bullets
5. Application message draft
6. Final ATS-safe checklist

### 5.6 · Interview Prep Pack

**Purpose:** Prepare client for scheduled interviews with role-specific rehearsal.

**Required sections:**
1. Interview context (date, format, interviewers)
2. Company and stack snapshot
3. 5 likely technical questions + answer frames
4. Behavioral STAR prompts
5. Candidate ask-back questions
6. 24h readiness checklist

### 5.7 · Offer Decision Brief

**Purpose:** Support accept/decline/negotiation decisions using a weighted rubric.

**Required sections:**
1. Offer summary table
2. Weighted scoring rubric
3. Risk and upside notes
4. Negotiation strategy
5. Final recommendation and next action

### 5.8 · Weekly Conversion Report

**Purpose:** Track funnel outcomes and expose execution bottlenecks.

**Required sections:**
1. Weekly KPI snapshot
2. Stage conversion table (applied -> interview -> offer -> accepted)
3. Follow-up compliance rate
4. False positive review and parameter adjustments
5. Next-week operating priorities

### 5.9 · Readiness Assessment Report

**Purpose:** Summarize technical, behavioral, and language readiness with a clear gate.

**Required sections:**
1. Assessment context and score summary
2. Domain-level scores (technical/behavioral/language)
3. Strengths and risk gaps
4. Improvement plan with deadlines
5. Readiness gate: `READY`, `PARTIAL`, `NOT READY`

## 6 · Quality Gate Checklist

Before publishing ANY client-facing document, verify:

- [ ] Can the client identify the top 3 actions within 30 seconds of opening?
- [ ] Are the best opportunities at the very top (not buried)?
- [ ] Are all links short markdown hyperlinks (not raw URLs)?
- [ ] Is there at least one interpretation paragraph after each data section?
- [ ] Are tables using proper alignment (center for #/tier, right for scores)?
- [ ] Are boolean values shown as ✅/❌ icons, not "Yes"/"No" text?
- [ ] Does the document have a header block with date and agency name?
- [ ] Does every H2 section have a `---` separator above or below it?
- [ ] Is the tone human, direct, and recruiter-like (not robotic)?
- [ ] Does the document end with a footer and source references?
- [ ] If A-tier jobs are presented, is there a requirement-to-client validation section?
- [ ] Does the document apply luxury readability patterns from `claude_luxury_format_universal.md`?
- [ ] If luxury patterns conflicted with template rules, did you keep this STYLE_GUIDE as authority?

**If any checkbox fails, the document is not ready. Fix it before delivering.**

## 7 · Anti-Patterns (Explicitly Prohibited)

These patterns are **BANNED** from all client-facing output:

| ❌ Anti-Pattern | Why It's Bad |
|:---|:---|
| Raw URLs in table cells | Destroys readability, breaks table width |
| "Yes" / "No" text for booleans | Icons (✅/❌) scan 10× faster |
| No interpretation after data | Client sees numbers but doesn't know what they mean |
| Plain numbered lists for action plans | Tables with Step/Action/Target are clearer |
| Missing section separators (`---`) | Document feels like one continuous wall |
| Headings without numbering | Client can't reference "section 3" in conversation |
| Starting a document without a header block | No context about when/who/what |
| Using "Here is the data" or "As requested" | Sounds like a bot, not an agent |
| Giant paragraphs (>4 lines) | Break them up. Short blocks only. |
| Missing emoji on H1 title | Feels flat. One emoji sets the tone. |
| A-tier recommendation without fit proof | Client cannot trust why role was selected |
